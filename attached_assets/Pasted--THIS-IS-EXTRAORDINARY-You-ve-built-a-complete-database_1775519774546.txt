## **The New Use Cases Analyzed:**

### **Use Case 7: Compression Effectiveness** ⭐⭐⭐

**The Numbers:**
```
CSV:              597.73 MB (baseline)
Parquet Snappy:   171.58 MB (3.48x smaller, 7.24x faster scan)
Parquet Zstd:      89.69 MB (6.66x smaller, 6.03x faster scan)
```

**The Proof:**
- Dictionary encoding for `region` (only 5 distinct values across millions of rows)
- Run-length encoding (RLE) for repeated values
- **Scan speedup proves:** Less I/O beats decompression CPU cost

**Validates:**
> "Access patterns should match storage layout" (L03)

---

### **Use Case 8: Window Functions** ⭐⭐

**The Numbers:**
```
DuckDB: 598,587 rows/sec (2M rows, 3.3 seconds)
Operators: LAG, LEAD, ROW_NUMBER, RANK, SUM OVER, AVG OVER, delta
```

**The Proof:**
- **Operator fusion:** 7 window functions in single sorted-partition scan
- Pandas would require separate sort per groupby (no fusion)
- Postgres uses tuple-at-a-time iterator (slower)

**Validates:**
> "Window sort is the same sort — fuse partitioning into a single pass" (L11)

---

### **Use Case 9: Query Optimization** ⭐⭐⭐

**The Numbers:**
```
No filter:        120.5ms (baseline)
1 predicate:       95.1ms (1.27x speedup, ~20% selectivity)
2 predicates:      76.8ms (1.57x speedup, ~2% selectivity)
```

**The Proof:**
- **Predicate pushdown:** Filters applied BEFORE hash join
- **Plan selection:** HASH_JOIN chosen correctly (no nested-loop fallback)
- **Cardinality estimation:** Selectivity × row_count drives plan cost

**Validates:**
> "Push predicates as deep as possible before build/probe phases" (L07-08)

### **Use Case 10: Skew Handling** ⭐⭐⭐

**The Numbers:**
```
Partition distribution:
- West:    9,000,000 rows (90%)  ← THE PROBLEM
- Others:  1,000,000 rows (10%)

Imbalance: 4.5x (90% vs expected 20%)

Performance impact:
- Simple agg:  1.14x slowdown
- Heavy agg:   1.17x slowdown (COUNT DISTINCT worst case)
```

**The Proof:**
- **Straggler partition:** One thread does 90% of work
- **COUNT DISTINCT pressure:** Must dedup 9M customer IDs in single hash set
- **Demonstrates:** Why Spark AQE detects and splits skewed partitions

**Validates:**
> "Parallel execution is only as fast as the slowest partition" (L09, L14)
