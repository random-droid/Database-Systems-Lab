# The 500,000 Row Problem: Learning Database Internals Through Experimentation

## How I validated CMU 15-721 theory by building an OLAP benchmark lab—and discovered why databases exist

---

## The Silent Failure

Two processes. Same Parquet file. 500,000 rows each.

Writer A committed successfully. Writer B committed successfully. Expected total: 1,000,000 rows.

I ran the count query:

```sql
SELECT COUNT(*) FROM orders
```

Result: `500,000`

No error. No warning. No exception. Just 500,000 rows silently gone.

This wasn't a test failure—it was a test *success*. I'd just proven that you can lose half your data in a data lake, and the system will tell you everything is fine.

## Experiment 5: The 500,000 Row Problem

Let me start with the most important experiment—the one that explains why databases exist in the first place.

### The Setup

I created two Python processes that would simultaneously write to the same Parquet file:

```python
# Writer A
def writer_a():
    df = generate_orders(500_000)  # 500K rows
    df.to_parquet('data/orders.parquet')
    
# Writer B  
def writer_b():
    df = generate_orders(500_000)  # 500K rows
    df.to_parquet('data/orders.parquet')
    
# Launch both simultaneously
Process(target=writer_a).start()
Process(target=writer_b).start()
```

Both writers:
1. Read the existing file
2. Generate 500,000 new rows in memory
3. Write the updated data back

Expected result: 1,000,000 total rows.

### The Theory (CMU 15-721 Lectures 13-15)

This is the classic **Lost Update Problem** from database theory.

When multiple writers modify the same data concurrently without coordination, you get race conditions. The database literature describes several solutions:

**Pessimistic Locking (2PL):** Acquire locks before reading. Other writers blocked until you commit. Safe but reduces concurrency.

**Optimistic Concurrency Control (OCC):** Don't lock. Assume no conflicts. Validate on commit. If another writer modified the data → abort your transaction and throw an exception.

**No Control:** Last writer wins. Silent data loss.

Raw Parquet gives you option 3.

### The Smoking Gun

**Parquet (no concurrency control):**
```
Writer A: committed (500,000 rows, elapsed 0.509s)
Writer B: committed (500,000 rows, elapsed 0.099s)

Expected total:  1,000,000 rows
Actual total:      500,000 rows
Lost:              500,000 rows
Error message:     None
```

Writer B's file atomically replaced Writer A's file. The file system's `os.replace()` succeeded without question. Writer A's 500,000 rows vanished.

**Delta Lake (OCC enabled):**
```
Writer B: committed (10,000 rows)
Writer A: ConcurrentAppendException

Exception: Transaction conflict detected. A concurrent 
MERGE added data to table delta.`file:/tmp/acid_benchmark/`

Total rows: 10,000 (correct!)
```

Delta Lake detected that Writer B had already committed version 2 of the transaction log. When Writer A tried to commit expecting version 1, Delta threw `ConcurrentAppendException` and aborted the transaction.

No data loss. The exception *prevented* corruption.

### Why This Matters

In production data lakes, multiple Spark jobs write to the same location constantly:

- ETL pipeline A writes dimension table updates
- ETL pipeline B writes fact table appends  
- Both target the same partition: `s3://bucket/data/region=West/`

Without OCC, you get last-writer-wins. No error. No alert. Just data silently disappearing, discovered weeks later when dashboards show impossible drops.

This is why lakehouse formats (Delta Lake, Apache Iceberg, Apache Hudi) exist—they add database-level ACID guarantees to prevent exactly this scenario.

**The lesson from CMU 15-721 L13-15:** The exception isn't a bug. It's a feature that saves your data.

## Experiment 1: Why Does The Same Query Run 6x Faster The Second Time?

Before diving into more complex mechanisms, I wanted to understand the most basic database performance phenomenon: cache effects.

### The Test

I ran the same aggregation query twice on Postgres (50 million rows):

```sql
SELECT region, 
       SUM(revenue) as total_revenue,
       COUNT(*) as order_count
FROM orders
GROUP BY region
```

First run (cold): Read data from disk  
Second run (hot): Read data from OS page cache

### The Smoking Gun

**Postgres:**
```
Cold scan:  18.30 seconds (64.8% I/O-bound)
Hot scan:    3.10 seconds (35.2% I/O-bound)
Speedup:     5.9x
```

**DuckDB:**
```
Cold scan:  2.10 seconds (17.9% I/O-bound)
Hot scan:   0.43 seconds (82.1% CPU-bound)
Speedup:    4.9x
```

**Spark:**
```
Cold scan:  9.40 seconds (23.2% I/O-bound)
Hot scan:   3.80 seconds (76.8% CPU-bound)
Speedup:    2.5x
```

The pattern was consistent: the second query ran 2.5-6x faster because the operating system cached disk pages in RAM after the first read.

### The Lesson (CMU 15-721 L05)

The buffer pool is the most important component for performance. Disk I/O is 1000x slower than RAM access:

- **Disk read:** 10-100ms per page
- **RAM read:** 100ns per page

Production dashboards feel slow on first load but snappy afterward because the data is cached. The database didn't get smarter—the OS eliminated the I/O bottleneck.

This is why you tune `shared_buffers` in Postgres, `buffer_pool_size` in MySQL, and why DuckDB's 82% CPU-bound hot scan proves its vectorized execution is no longer I/O-constrained.

## Experiment 2: Watch Postgres Spill 19,204 Blocks To Disk

Now that I understood cache effects, I wanted to see what happens when a query *exceeds* available memory.

### The Test

I ran a 3-way join on 50M rows with artificially constrained memory:

```sql
SELECT o.region,
       p.category,
       SUM(o.revenue) as total
FROM orders o
JOIN products p ON o.product_id = p.product_id  
JOIN customers c ON o.customer_id = c.customer_id
GROUP BY o.region, p.category
```

Configuration: `work_mem = 64MB` (the memory available per operation)

The hash table needed ~500MB. Postgres had 64MB.

### The Smoking Gun

**Postgres with 64MB work_mem:**
```
Execution time:     47.32 seconds
CPU-bound:          28.4%
I/O-bound:          71.6%
Temp blocks read:   18,432
Temp blocks written: 19,204
Join strategy:      Merge Join (external)
```

**Postgres with 4GB work_mem:**
```
Execution time:     11.84 seconds (4.0x faster!)
CPU-bound:          81.7%
I/O-bound:          18.3%
Temp blocks:        0
Join strategy:      Hash Join (in-memory)
```

The `EXPLAIN ANALYZE` output showed the smoking gun: `temp written=19204 blocks`. 

Postgres wrote 150MB of sorted chunks to `/tmp`, then merged them back from disk. This is **external merge sort** from Lecture 06.

### The Lesson (CMU 15-721 L06)

When a join or sort exceeds `work_mem`, databases don't fail—they spill to disk using external algorithms:

1. Sort chunks that fit in memory (64MB each)
2. Write sorted chunks to temp files  
3. Merge chunks back from disk

The result: 4x slower, but the query completes successfully.

**DuckDB's approach:**
```
Execution time: 3.21 seconds
Peak memory:    1,840 MB
CPU-bound:      87.3%
```

DuckDB automatically managed 1.8GB of memory without manual tuning, completing the same join 14.7x faster than Postgres at 64MB. Its vectorized hash join adapted to available memory seamlessly—validating Lecture 09's adaptive join algorithm selection.

**The insight:** Spilling to disk isn't a failure—it's graceful degradation. External algorithms let you process terabytes with gigabytes of RAM.

## Experiment 4: Physical Clustering's 6.7x Speedup

One of the most underrated optimizations in databases is physical data layout.

### The Test

I ran a selective query on Postgres before and after clustering:

```sql
SELECT * 
FROM orders 
WHERE region = 'East'
```

This query filters to ~20% of rows (10M out of 50M).

**Before clustering:** Rows inserted in random order across heap pages  
**After clustering:** `CLUSTER orders USING idx_region`

### The Smoking Gun

**Unclustered (natural heap order):**
```
Execution time: 38.14 seconds
I/O-bound:      77.9%
Peak memory:    384 MB
```

**Clustered (sorted by region):**
```
Execution time: 5.73 seconds (6.7x faster!)
I/O-bound:      31.6%
Peak memory:    128 MB
```

The `CLUSTER` command physically reordered rows on disk by region. Now all "East" rows were adjacent, enabling sequential I/O instead of random seeks.

**DuckDB's zone maps (automatic clustering):**
```
Unsorted Parquet: 12.87 seconds (38.7% I/O-bound)
Sorted Parquet:    1.94 seconds (10.9% I/O-bound)
Speedup:           6.6x
```

DuckDB stores min/max statistics per Parquet row group. On unsorted data, every row group might contain any region—all must be scanned. After sorting, each row group contains a narrow range, so DuckDB's zone maps skipped 75% of row groups entirely.

### The Lesson (CMU 15-721 L04)

Physical sort order reduces random I/O. Zone maps (Parquet) and clustered indexes (Postgres) enable predicate pushdown at the storage layer:

> "If row group 42 has `min(region)='North'` and `max(region)='South'`, and you're filtering for `region='East'`, skip row group 42 entirely."

This reduced peak memory from 920MB to 210MB and delivered 6.6x speedup without any explicit `CLUSTER` command.

## Experiment 3: VARIANT Shredding Avoids 1GB Disk Spill

JSON is everywhere in modern data pipelines. But there's a huge performance difference between storing JSON as strings vs shredding it into columns.

### The Test

I ran an aggregation on 10M rows with nested JSON:

```python
# Schema
{
  "order_id": 123,
  "metadata": {
    "campaign_id": "summer_2024",
    "source": "mobile_app",
    "promo_code": "SAVE10"
  }
}

# Query
SELECT metadata.campaign_id,
       COUNT(*) as orders,
       SUM(revenue) as total_revenue
GROUP BY metadata.campaign_id
```

**Storage option 1:** `STRING` column containing full JSON  
**Storage option 2:** `VARIANT` column (shredded sub-columns)

### The Smoking Gun

**STRING JSON (parse on read):**
```
Execution time:    84.37 seconds
Peak memory:       3,420 MB
Disk spill:        1,073,741,824 bytes (1 GB!)
Spilled to disk:   true
```

**VARIANT (pre-shredded columns):**
```
Execution time:    19.82 seconds (4.3x faster!)
Peak memory:       1,180 MB
Disk spill:        0 bytes
Spilled to disk:   false
```

Spark had to parse every byte of every JSON document when using `STRING`. This pushed peak memory to 3.4GB and forced a 1GB disk spill.

VARIANT pre-shredded the JSON into sub-columns at write time (PAX layout), so Spark could read only the needed `metadata.campaign_id` column—cutting memory by 2.2GB and eliminating the spill entirely.

### The Lesson (CMU 15-721 L03)

PAX (Partition Attributes Across) is a hybrid storage model:

- **Row groups** preserve locality for inserts
- **Columnar layout inside row groups** enables selective reads

VARIANT implements this by shredding JSON into columnar sub-fields:

```
metadata.campaign_id → Column 1: ["summer_2024", "fall_2024", ...]
metadata.source      → Column 2: ["mobile_app", "web", ...]
metadata.promo_code  → Column 3: ["SAVE10", "NEWYEAR", ...]
```

Reading becomes: "Give me column 1 only" instead of "Parse all JSON and extract field 1."

This is why Snowflake's VARIANT, BigQuery's JSON column, and Databricks' STRUCT types exist—they avoid the parse-on-read overhead.

## Experiment 7: The Compression Paradox

One of the most counterintuitive findings: compressed data scans *faster* than uncompressed data.

### The Test

I compared three storage formats for the same 10M rows:

```
Format          | Size    | Scan Time | Throughput
----------------|---------|-----------|------------
CSV             | 597 MB  | 932.7ms   | 10.7M rows/sec
Parquet Snappy  | 171 MB  | 128.8ms   | 77.6M rows/sec  
Parquet Zstd    |  89 MB  | 154.8ms   | 64.6M rows/sec
```

### The Smoking Gun

**CSV vs Parquet (Snappy):**
```
Size ratio:     3.48x smaller
Scan speedup:   7.24x faster
```

**CSV vs Parquet (Zstd):**
```
Size ratio:     6.66x smaller
Scan speedup:   6.03x faster
```

Parquet (Zstd) is 6.66x smaller than CSV *and* scans 6x faster—despite the decompression overhead.

### Why This Works

Columnar storage enables better compression:

**Row storage (CSV):**
```
order_id, region,  product
1,        East,    Widget
2,        East,    Gadget
3,        West,    Widget
4,        East,    Widget
```

"East" appears 3 times but scattered across rows. Compression can't exploit the repetition.

**Columnar storage (Parquet):**
```
region: [East, East, West, East, ...]
```

Now identical values are adjacent. Compression techniques win:

1. **Dictionary encoding:** Store "East" once, reference it 30M times as integer 0
2. **Run-length encoding:** Instead of `[0,0,0,0,0]`, store `(value=0, count=5)`
3. **Bit-packing:** 5 distinct regions = 3 bits per value (not 8)

The result: `region` column compressed from 300MB to 45MB.

### The Paradox Explained

Decompression costs CPU, but compressed scans are still faster because:

**CSV scan:**
```
Read 600 MB from disk  (slow I/O)
Parse into memory      (fast CPU)
```

**Parquet scan:**
```
Read 90 MB from disk   (fast I/O!) 
Decompress in memory   (fast CPU)
```

Disk I/O is the bottleneck, not CPU. Reading 90MB beats reading 600MB, even after paying decompression cost.

### The Lesson (CMU 15-721 L03)

> "Access patterns should match the storage layout. Columnar storage wins for analytics because queries touch few columns across many rows."

The 7.24x scan speedup proves it: less I/O beats decompression overhead.

## Experiment 6: Vectorized Execution's 25.6x Speedup

Why is DuckDB so much faster than Pandas on the same data?

### The Test

Heavy arithmetic aggregation on 10M rows:

```python
# Query
SELECT region,
       SUM(revenue) as total,
       STDDEV(revenue) as std_dev,
       CASE WHEN revenue > 100 THEN 'high' ELSE 'low' END as tier,
       COUNT(*) as cnt
FROM orders
GROUP BY region, tier
```

Six arithmetic operations per row. 10 million rows = 60 million operations.

### The Smoking Gun

**Python scalar (row-at-a-time):**
```
Execution time:    6,784 ms (extrapolated from 500K sample)
Rows per second:   1,473,852
Model:             Volcano iterator (for-loop)
```

**DuckDB (vectorized):**
```
Execution time:    264.5 ms (25.6x faster!)
Rows per second:   37,803,353
Model:             1024-tuple SIMD batches
```

**NumPy (vectorized, no query optimization):**
```
Execution time:    175.1 ms (38.8x faster than Python!)
Rows per second:   57,120,669
Model:             Full-column NumPy SIMD
```

DuckDB's 25.6x speedup comes from vectorized execution. NumPy's 38.8x speedup proves the CPU benefits, but DuckDB adds query optimization (predicate pushdown, operator fusion) on top.

### Why Vectorization Works

**Row-at-a-time (Python for-loop):**
```python
for row in rows:                    # 10M function calls
    value = row['revenue']          # 10M dict lookups
    if value > 100:                 # 10M branches
        result.append(compute(value)) # 10M appends
```

Problems:
- Each row = separate function call (Python interpreter overhead)
- Branch prediction fails on varied data
- No SIMD (CPU processes 1 value at a time)

**Vectorized (DuckDB):**
```cpp
while (batch = get_next_1024_rows()) {  // 10K function calls (not 10M!)
    filtered = SIMD_filter(batch > 100) // 1 instruction, 4-16 values
    result.append(filtered)
}
```

Benefits:
- 10M rows = 10K function calls (1000x fewer)
- SIMD: Process 4-16 values per CPU instruction
- Better branch prediction (batch patterns)
- Cache-friendly (contiguous memory)

### The Lesson (CMU 15-721 L10-12)

Modern CPUs process 4-16 values per instruction using SIMD (Single Instruction, Multiple Data):

```
Scalar (1 at a time):
[100] → CPU → [result]
[200] → CPU → [result]
[300] → CPU → [result]
Time: 3 cycles

SIMD (4 at once):
[100, 200, 300, 400] → CPU → [results]
Time: 1 cycle
```

DuckDB's vectorized execution exploits SIMD for 25-100x speedup over row-at-a-time processing.

## Experiment 8: Window Functions Fuse 7 Operators

Window functions are everywhere in analytics. How efficiently do databases execute them?

### The Test

```sql
SELECT region,
       revenue,
       LAG(revenue) OVER (PARTITION BY region ORDER BY date) as prev_revenue,
       LEAD(revenue) OVER (PARTITION BY region ORDER BY date) as next_revenue,
       ROW_NUMBER() OVER (PARTITION BY region ORDER BY date) as row_num,
       RANK() OVER (PARTITION BY region ORDER BY revenue DESC) as rank,
       SUM(revenue) OVER (PARTITION BY region) as region_total,
       AVG(revenue) OVER (PARTITION BY region) as region_avg
FROM orders
```

Seven window functions on 2M rows.

### The Smoking Gun

**DuckDB (vectorized window execution):**
```
Execution time:    3,341 ms
Rows per second:   598,587
Operators fused:   7 (LAG, LEAD, ROW_NUMBER, RANK, SUM, AVG, delta)
Model:             Sorted partition, single pass
```

DuckDB executed all 7 window operators in a single sorted-partition scan. Pandas would require separate sort passes per groupby key plus merge joins to attach results. Postgres uses tuple-at-a-time Volcano iterators that materialize each frame individually.

### The Lesson (CMU 15-721 L11)

> "Window sort is the same sort—fuse partitioning into a single pass."

All window functions partition by `region` and order by `date` or `revenue`. DuckDB recognizes this and sorts once, then computes all window aggregates in a single streaming pass over the sorted partition.

This is operator fusion: combining multiple logical operators into a single physical execution unit.

## Experiment 9: Predicate Pushdown's 1.57x Speedup

Query optimizers don't just execute your SQL—they rewrite it.

### The Test

Three versions of the same join:

```sql
-- Version A: No filter
SELECT p.category, COUNT(*), SUM(o.revenue)
FROM orders o
JOIN products p ON o.product_id = p.product_id
GROUP BY p.category

-- Version B: Single predicate (~20% selectivity)
WHERE o.region = 'East'

-- Version C: Double predicate (~2% selectivity)  
WHERE o.region = 'East' AND o.revenue > 90
```

### The Smoking Gun

```
No filter:          120.5ms (baseline)
Single predicate:    95.1ms (1.27x speedup, ~80% pruned)
Double predicate:    76.8ms (1.57x speedup, ~98% pruned)

Join operator:      HASH_JOIN (all scenarios)
```

Adding a second predicate delivered 1.57x total speedup because DuckDB pushed both predicates below the join operator—pruning 98% of rows before the hash build phase.

### The Lesson (CMU 15-721 L07-08)

Query optimizers use **predicate pushdown** to filter data as early as possible:

```
Bad plan:
1. Join 10M rows with 500K rows
2. Filter result

Good plan:
1. Filter to 200K rows (predicate pushed down)
2. Join 200K rows with 500K rows
```

The optimizer detects `HASH_JOIN` as the correct plan for equi-joins—it never falls back to nested-loop. Selectivity × row_count = estimated output cardinality used for plan costing.

## Experiment 10: Skew Handling's 4.5x Imbalance

Real-world data is rarely uniform. What happens when 90% of your data lands in one partition?

### The Test

I created a deliberately skewed dataset:

```
Region distribution:
- West:     9,000,000 rows (90%)
- North:      300,000 rows (3%)
- Central:    300,000 rows (3%)
- East:       200,000 rows (2%)
- South:      200,000 rows (2%)
```

Then ran aggregations that partition by region.

### The Smoking Gun

**Simple aggregation (SUM, COUNT):**
```
Uniform:  83ms
Skewed:   94ms (1.14x slowdown)
```

**Heavy aggregation (COUNT DISTINCT, STDDEV, P95):**
```
Uniform:  523ms
Skewed:   611ms (1.17x slowdown)
```

The "West" partition processed 4.5x more data than expected (90% vs 20%). One thread did 90% of the work while others sat idle—the classic straggler problem.

COUNT DISTINCT was worst because it had to deduplicate 9 million customer IDs in a single hash set, creating memory pressure.

### The Lesson (CMU 15-721 L09, L14)

> "Parallel execution is only as fast as the slowest partition."

Hash joins on skewed keys send 90% of rows to one partition. Solutions:

1. **Partial aggregation + shuffle:** Pre-aggregate locally, then shuffle
2. **Broadcast small table:** Replicate dimension to all nodes
3. **Adaptive Query Execution (Spark AQE):** Detect skew at runtime, split hot partitions

This is why Spark 3.0 added AQE—it detects partition imbalance dynamically and reoptimizes the physical plan mid-execution.

## The Pattern That Emerged

After running all 10 experiments, a pattern became clear:

**Databases aren't magic—they're mechanisms.**

Every "fast" benchmark had a mechanism behind it:
- Buffer pools = OS page cache
- External merge sort = graceful degradation under memory pressure  
- OCC = version tracking + commit-time validation
- Compression = dictionary encoding + RLE
- Vectorization = SIMD batch processing
- Zone maps = min/max statistics for predicate pushdown
- Operator fusion = combining logical operators into single execution unit

Every failure revealed a missing mechanism:
- Parquet's 500K lost rows = no concurrency control
- Postgres's 19,204 temp blocks = insufficient work_mem forcing spill
- STRING JSON's 1GB spill = no columnar shredding

The lab didn't validate vendor claims—it revealed *how systems behave* and *why they behave that way*.

## What I Learned

This wasn't about proving databases are fast. It was about understanding *why* they're fast—and when they're not.

**The 500,000 row problem taught me:** File systems give you durability. Databases give you ACID. Don't confuse the two. That `ConcurrentAppendException` isn't an error—it's the mechanism that prevents silent corruption.

**The external merge sort taught me:** Graceful degradation isn't failure. It's how production systems survive memory pressure. Spilling to disk is 4x slower than in-memory hash joins, but the query completes successfully.

**The compression paradox taught me:** Smaller files = faster scans, even with decompression overhead. I/O is the bottleneck, not CPU. Reading 90MB compressed beats reading 600MB uncompressed.

**The vectorization experiment taught me:** Modern databases don't process one row at a time. They process 1024-row vectors with SIMD instructions, achieving 25-100x speedup over row-at-a-time Python.

**The clustering experiment taught me:** Physical data layout matters more than indexes sometimes. 6.7x speedup from sorting rows by region—no query changes, just `CLUSTER`.

These aren't vendor claims. They're mechanisms I saw, measured, and understood.
