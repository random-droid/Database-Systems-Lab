## **Reframed Theory Cards:**

### **Instead of "What to Look For" → "What You'll Discover"**

```jsx
<TheoryCard>
  <Title>Experiment: Concurrency Control</Title>
  <Subtitle>What happens when two writers collide?</Subtitle>
  
  <TheQuestion>
    <h4>🤔 The Question:</h4>
    <p>
      You have two processes trying to write to the same file 
      simultaneously. What happens?
    </p>
  </TheQuestion>
  
  <TheExperiment>
    <h4>🧪 The Experiment:</h4>
    <ol>
      <li>Writer A starts modifying 500K rows</li>
      <li>Writer B starts modifying the SAME 500K rows</li>
      <li>Both try to commit at the same time</li>
      <li>Click "Run Experiment" and see what happens</li>
    </ol>
  </TheExperiment>
  
  <WhatYoullDiscover>
    <h4>💡 What You'll Discover:</h4>
    
    <Discovery type="parquet">
      <strong>With Parquet (no concurrency control):</strong>
      <ul>
        <li>Both writers think they succeeded ✅</li>
        <li>Check the file → only 500K rows (should be 1M!)</li>
        <li>No error, no warning, just silent data loss</li>
        <li><strong>WHY:</strong> File system atomic rename = last writer wins</li>
      </ul>
    </Discovery>
    
    <Discovery type="delta">
      <strong>With Delta Lake (OCC enabled):</strong>
      <ul>
        <li>Writer B commits successfully ✅</li>
        <li>Writer A gets: ConcurrentAppendException ❌</li>
        <li>All data intact (500K rows from B)</li>
        <li><strong>WHY:</strong> Delta detected version conflict and aborted Writer A</li>
      </ul>
    </Discovery>
  </WhatYoullDiscover>
  
  <TheLesson>
    <h4>📚 The Lesson (CMU 15-721 L14):</h4>
    <p>
      This is <strong>Optimistic Concurrency Control (OCC)</strong> in action.
      Instead of locking the file (slow), Delta Lake:
    </p>
    <ol>
      <li>Lets both writers proceed without blocking</li>
      <li>Tracks what version each writer started from</li>
      <li>On commit, validates version hasn't changed</li>
      <li>If changed → abort transaction (prevents corruption)</li>
    </ol>
    <p>
      <strong>Key insight:</strong> Throwing an exception is a FEATURE, not a bug—
      it prevents silent data loss!
    </p>
  </TheLesson>
  
  <TryItYourself>
    <h4>🎮 Try It Yourself:</h4>
    <ul>
      <li>Run the experiment multiple times</li>
      <li>Watch the execution logs</li>
      <li>See the exception stack trace</li>
      <li>Count the final rows</li>
    </ul>
  </TryItYourself>
</TheoryCard>
```

          </ul>
        </Discovery>
      </DiscoverySection>

        <h3>💡 Why This Happens:</h3>
        
        <h4>Parquet (No Concurrency Control):</h4>
        <CodeExample>
          Writer A creates: part-00001.parquet (500K rows)
          Writer B creates: part-00001.parquet (500K rows)
          
          File system: "Two files with same name!"
          → Atomic rename: Last writer wins
          → Writer B's file overwrites Writer A's file
          → No error (atomic rename always succeeds)
          
          Result: Silent data loss ❌
        </CodeExample>
        
        <h4>Delta Lake (Optimistic Concurrency Control):</h4>
        <CodeExample>
          Writer A reads log → version 1
          Writer B reads log → version 1
          
          Writer B commits:
          → Writes part-00002.parquet
          → Updates _delta_log/00002.json
          → Success! ✅
          
          Writer A tries to commit:
          → Expected version: 2
          → Current version: 2 (B already committed!)
          → ConcurrentAppendException ❌
          → Transaction aborted
          
          Result: Conflict detected, no data loss ✅
        </CodeExample>
        
        <h4>The Lesson (CMU 15-721 L13-15):</h4>
        <p>
          This is why databases exist! File systems give you durability,
          but databases give you ACID. Optimistic Concurrency Control
          (OCC) allows high concurrency while preventing corruption.
        </p>
      </TheLesson>
      
      <RealWorldImpact>
        <h3>🌍 Why This Matters:</h3>
        <p>
          In production data pipelines, multiple Spark jobs often write
          to the same location. Without OCC:
        </p>
        <ul>
          <li>ETL job overwrites another ETL job's output</li>
          <li>No error, no alert</li>
          <li>Data silently disappears</li>
          <li>Discovered weeks later in dashboards</li>
        </ul>
        <p>
          This is why lakehouse formats (Delta/Iceberg/Hudi) exist—
          they add database-level guarantees to data lakes.
        </p>
      </RealWorldImpact>
    </Experiment>
